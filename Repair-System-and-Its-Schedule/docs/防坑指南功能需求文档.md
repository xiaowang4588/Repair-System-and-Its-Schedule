> 版本：v1.0

## 一、项目背景

### 1.1 当前痛点

学校多媒体运维团队在日常设备维修工作中，积累了大量宝贵的实战经验，但这些经验目前存在以下问题：

1. **经验散落**：维修经验分散在微信群聊记录中，难以查找和复用
2. **重复踩坑**：新人遇到相同故障时，不知道前人是如何解决的，重复摸索
3. **记录成本高**：传统的"技术文档"形式枯燥乏味，没人愿意主动记录
4. **缺乏检索**：无法按设备类型、故障现象等维度快速检索历史经验

### 1.2 解决方案

在现有报修系统的学生端中，新增"防坑指南"模块——一个类"微信朋友圈"形式的轻量化经验分享与互助平台。

**核心价值**：降低记录成本，把生硬的"技术文档"变成随手发的"朋友圈动态"，让团队经验可沉淀、可搜索，帮助后人少走弯路。

### 1.3 与现有系统的关系

本功能**复用**现有报修系统的以下基础设施：

| 复用项 | 来源 | 用途 |
|--------|------|------|
| 学生账号体系 | `Student` 模型 + HMAC Token 认证 | 登录态复用，无需额外注册 |
| 故障类型标签 | `fault_type` 枚举（中控/电脑/投影仪等） | 作为防坑指南的设备标签 |
| 教室/楼栋数据 | `data_cleaning.py` 课表数据 | 作为地点标签来源 |
| 图片上传接口 | `/api/repair/upload-image` | 复用已有图片上传能力 |
| 个人中心页面 | `pages/profile/profile.vue` | 扩展"我的收藏"入口 |
| 首页入口 | `pages/index/index.vue` | 新增防坑指南快捷入口 |

## 二、功能模块设计

### 2.1 整体架构

```
学生端 App
├── 首页（新增"防坑指南"快捷入口）
├── 防坑指南页面（新增）
│   ├── 顶部：全局搜索框（搜设备、故障现象、发言人）
│   ├── 中部：标签快捷筛选通道（#投影仪、#电脑、#达者楼...）
│   └── 下部：朋友圈动态流
│        ├── 每条动态：文字 + 图片 + 标签 + 点赞/评论/收藏
│        └── 悬浮按钮：[发布动态]
├── 个人中心（扩展）
│   ├── 我的发布（自己发过的防坑指南）
│   └── 我的收藏（收藏的高价值内容）
└── 发布页面（新增）
    ├── 文字输入
    ├── 九宫格图片上传（最多9张）
    ├── 设备标签选择（可选）
    └── 地点标签选择（可选）
```

### 2.2 动态发布（"发朋友圈"）

让维修人员在现场能以最快、最省力的方式记录问题和解决方案。

#### 2.2.1 发布内容

| 内容类型 | 说明 | 是否必填 | 限制 |
|----------|------|----------|------|
| 文字描述 | 故障现象、解决过程、经验总结 | ✅ 必填 | 最少5字，最多1000字 |
| 图片 | 故障照片、拆卸过程、接线图示 | 可选 | 最多9张，单张≤10MB，支持jpg/png/gif/webp |
| 视频 | 故障演示、拆卸过程、操作教程 | 可选 | 最多1个，时长≤60秒，大小≤50MB，支持mp4/mov |
| 设备标签 | 关联故障设备类型 | ✅ 必填 | 至少选1个，最多选3个 |
| 地点标签 | 关联故障发生楼栋/教室 | 可选 | 最多选1个 |

> **图片与视频的关系**：图片和视频互不影响，可同时上传。一条动态最多包含 9 张图片 + 1 个视频。

#### 2.2.2 标签体系

**设备标签**（复用现有 `fault_type` 枚举）：

| 标签 | 对应现有字段值 | 说明 |
|------|---------------|------|
| #中控 | fault_type: 中控 | 中控系统相关故障 |
| #电脑 | fault_type: 电脑 | 电脑主机相关故障 |
| #投影仪 | fault_type: 投影仪 | 投影仪相关故障 |
| #音响 | fault_type: 音响 | 音响设备相关故障 |
| #麦克风 | fault_type: 麦克风 | 麦克风相关故障 |
| #展台 | fault_type: 展台 | 展台相关故障 |
| #幕布 | fault_type: 幕布 | 幕布相关故障 |
| #网络 | fault_type: 网络 | 网络相关故障 |
| #软件 | fault_type: 软件 | 软件系统相关故障 |
| #其他 | fault_type: 其他 | 其他类型故障 |

**地点标签**（复用现有楼栋数据）：

| 标签 | 来源 | 说明 |
|------|------|------|
| #行者楼 | 课表数据中的楼栋名 | 行者楼区域 |
| #达者楼 | 课表数据中的楼栋名 | 达者楼区域 |
| #智者楼 | 课表数据中的楼栋名 | 智者楼区域 |
| #仁者楼 | 课表数据中的楼栋名 | 仁者楼区域 |
| #其他 | - | 其他地点 |

> 地点标签列表从 `/api/buildings` 接口动态获取，无需硬编码。

#### 2.2.3 发布流程

```
点击"发布"按钮
    → 进入发布页面
    → 输入文字描述（必填）
    → 上传图片（可选，最多9张，复用已有上传接口）
    → 录制/选择视频（可选，最多1个，≤60秒）
    → 选择设备标签（必选，至少1个，复用 fault_type 枚举）
    → 选择地点标签（可选，从楼栋列表获取）
    → 点击"发布"
    → 发布成功，返回信息流页面
```

### 2.4 智能搜索与标签聚合

这是区别于普通微信朋友圈最核心的功能，让动态变成"查阅手册"。

#### 2.4.1 全局搜索

- 位于防坑指南页面顶部
- 支持输入关键词检索所有历史动态
- 搜索范围：文字内容、设备标签名、地点标签名、发布者姓名
- 搜索结果按时间倒序排列
- 支持搜索历史记录（最近10条）

- 点击任意标签（如 #投影仪），进入标签聚合页
- 该页面展示所有包含该标签的防坑指南动态
- 顶部显示标签名称和关联的动态总数
- 支持按时间排序（默认倒序）

## 三、数据库设计

### 3.1 新增数据表

使用 Peewee ORM，与现有 `repair.db` 数据库共用。

#### 3.1.1 GuidePost（防坑指南动态表）

```python
class GuidePost(Model):
    """防坑指南动态"""
    id = AutoField(primary_key=True)
    student_id = CharField(max_length=50, index=True)       # 发布者学号（关联 Student 表）
    student_name = CharField(max_length=50)                  # 发布者姓名（冗余存储，避免频繁JOIN）
    content = TextField()                                    # 文字内容（10-1000字）
    images = TextField(default='[]')                         # 图片路径列表（JSON数组，最多9张）
    video_url = CharField(max_length=500, default='')        # 视频路径（最多1个，≤60秒）
    video_duration = IntegerField(default=0)                 # 视频时长（秒）
    device_tags = CharField(max_length=200)                  # 设备标签（JSON数组，如 ["投影仪","中控"]）
    location_tag = CharField(max_length=50, default='')      # 地点标签（如 "行者楼"）
    like_count = IntegerField(default=0)                     # 点赞数（冗余计数，提升查询性能）
    comment_count = IntegerField(default=0)                  # 评论数
    favorite_count = IntegerField(default=0)                 # 收藏数
    is_deleted = BooleanField(default=False)                 # 软删除标记
    created_at = DateTimeField(default=datetime.now)         # 发布时间
    updated_at = DateTimeField(default=datetime.now)         # 更新时间

    class Meta:
        table_name = 'guide_post'
        order_by = ('-created_at',)
```

#### 3.1.2 GuideLike（点赞表）

```python
class GuideLike(Model):
    """防坑指南点赞"""
    id = AutoField(primary_key=True)
    post_id = IntegerField(index=True)                       # 关联动态ID
    student_id = CharField(max_length=50, index=True)        # 点赞者学号
    created_at = DateTimeField(default=datetime.now)         # 点赞时间

    class Meta:
        table_name = 'guide_like'
        indexes = (
            (('post_id', 'student_id'), True),  # 唯一索引，防止重复点赞
        )
```

#### 3.1.3 GuideComment（评论表）

```python
class GuideComment(Model):
    """防坑指南评论"""
    id = AutoField(primary_key=True)

    reply_to_id = IntegerField(null=True)                    # 回复的评论ID（一级回复）
    reply_to_name = CharField(max_length=50, default='')     # 被回复者姓名
    is_deleted = BooleanField(default=False)                 # 软删除标记
    created_at = DateTimeField(default=datetime.now)         # 评论时间

    class Meta:
        table_name = 'guide_comment'
        order_by = ('created_at',)
```

### 3.2 ER 关系图

```
Student (现有)
    │
    ├── 1:N ──→ GuidePost（一个学生发布多条动态）
    ├── 1:N ──→ GuideLike（一个学生点赞多条动态）
    ├── 1:N ──→ GuideComment（一个学生发多条评论）
    └── 1:N ──→ GuideFavorite（一个学生收藏多条动态）

GuidePost
    ├── 1:N ──→ GuideLike（一条动态被多人点赞）
    ├── 1:N ──→ GuideComment（一条动态有多条评论）
    └── 1:N ──→ GuideFavorite（一条动态被多人收藏）
```

## 四、API 接口设计

### 4.1 接口总览

所有接口复用现有的 HMAC Token 认证机制（`student_required` 装饰器）和响应格式规范。

| 接口 | 方法 | 认证 | 说明 |
|------|------|------|------|
| `/api/guide/create` | POST | ✅ | 发布防坑指南 |
| `/api/guide/list` | GET | 可选 | 获取动态列表（信息流） |
| `/api/guide/detail` | GET | 可选 | 获取动态详情 |
| `/api/guide/update` | POST | ✅ | 编辑自己的动态 |
| `/api/guide/delete` | POST | ✅ | 删除自己的动态 |
| `/api/guide/search` | GET | 可选 | 搜索动态 |
| `/api/guide/tags` | GET | 可选 | 获取所有标签及数量 |
| `/api/guide/like` | POST | ✅ | 点赞/取消点赞 |
| `/api/guide/comment` | POST | ✅ | 发表评论 |
| `/api/guide/comment/list` | GET | 可选 | 获取评论列表 |
| `/api/guide/comment/delete` | POST | ✅ | 删除自己的评论 |
| `/api/guide/favorite` | POST | ✅ | 收藏/取消收藏 |
| `/api/guide/my-posts` | GET | ✅ | 获取我的发布 |

| 查看标签聚合 | ✅ | ✅ | 公开可读 |
| 发布动态 | ❌ | ✅ | 需要登录 |
| 点赞 | ❌ | ✅ | 需要登录 |
| 评论 | ❌ | ✅ | 需要登录 |
| 收藏 | ❌ | ✅ | 需要登录 |
| 编辑动态 | ❌ | ✅ | 仅限自己的动态 |
| 删除动态 | ❌ | ✅ | 仅限自己的动态 |

### 7.2 安全措施

| 措施 | 说明 |
|------|------|
| Token 认证 | 复用现有 HMAC-SHA256 Token 机制 |
| 权限校验 | 编辑/删除操作校验 `student_id` 是否为发布者 |
| 内容过滤 | 文字内容去除HTML标签，防止XSS |
| 频率限制 | 同一用户每分钟最多发布3条动态、10条评论 |
| 图片校验 | 复用现有图片上传的MIME类型和大小限制 |
| 视频校验 | 校验视频格式（mp4/mov）、大小（≤50MB）、时长（≤60秒） |
| 软删除 | 动态和评论采用软删除，保留数据完整性 |
