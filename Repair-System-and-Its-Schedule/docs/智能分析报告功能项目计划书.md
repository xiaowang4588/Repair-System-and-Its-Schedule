## 一、项目概述

### 1.1 项目背景

当前系统的数据导出功能仅支持**原始数据导出**（Excel 表格），缺乏对数据的深度分析和智能建议。管理人员需要手动分析数据才能发现问题和趋势，效率低下。

**特别说明**：由于报修填写时间不等于实际报修时间（教师可能晚上统一填写），因此**不分析响应时间相关指标**。

### 1.2 项目目标

开发**智能分析报告**功能，实现：

1. **自动化分析**：系统自动分析报修数据，生成结构化报告
2. **周报重点**：详细展示本周报修情况，与上周对比
3. **教室维度**：聚焦具体教室的报修详情和历史
4. **智能建议**：基于数据分析结果，自动生成改进建议
5. **多格式导出**：支持 Excel、PDF、HTML 格式导出

### 1.3 核心价值

| 价值点 | 说明 |
|--------|------|
| 📊 数据驱动决策 | 用数据说话，而非凭经验判断 |
| ⏱️ 提升效率 | 自动生成报告，节省人工分析时间 |
| 🔍 发现问题 | 自动识别异常数据和潜在风险 |
| 📈 周对比 | 清晰了解每周变化趋势 |

## 二、需求分析

| 角色 | 需求 |
|------|------|
| 运维团队负责人 | 了解本周运维状况，制定下周计划 |

#### 2.2.3 分析维度（不含人员分析）

**基础统计**：
- 报修总量、待处理数、已处理数、处理率
- 较上周期对比（环比）

**故障分析**：
- 故障类型分布（饼图）
- 故障趋势变化（折线图）
- 高频故障 Top10
- 故障原因归类

**区域分析**：
- 楼栋报修分布（柱状图）
- 楼层报修热力图
- 高频故障教室 Top10
- 教室报修详情（含历史记录）

**时间分析**：
- 按周几分布（发现规律）
- 按节次分布（高峰时段）
- 按月份分布（季节性规律）

**学院分析**：
- 各学院报修数量
- 外聘教师报修占比
- 学院故障类型偏好

#### 2.2.4 智能建议

系统根据分析结果自动生成建议，包括：

| 分析结果 | 建议类型 | 示例 |
|----------|----------|------|
| 某教室反复报修 | 设备检查建议 | "行者楼408本学期报修8次，建议安排设备检查或更换" |
| 某类故障占比过大 | 专项培训建议 | "中控故障占比35%，建议组织中控使用培训" |
| 某楼栋故障率异常高 | 区域排查建议 | "行者楼4楼故障率高于平均值30%，建议安排专项检查" |
| 某时段报修集中 | 人员调配建议 | "周一上午报修量是平时的2倍，建议增派人手" |
| 待处理积压过多 | 流程优化建议 | "当前待处理20件，建议优先处理超过3天的工单" |
| 本周报修量激增 | 趋势预警 | "本周报修量较上周增长50%，需关注设备状况" |

### 3.2 核心模块设计

#### 3.2.1 数据分析器 (ReportAnalyzer)

```python
class ReportAnalyzer:
    """报告数据分析器"""

    def analyze(self, range_type: str, week_number: int = None) -> dict:
        """
        执行全面分析
        :param range_type: week/month/semester
        :param week_number: 周次（周报时必填）
        """
        if range_type == 'week':
            return self._analyze_weekly(week_number)
        elif range_type == 'month':
            return self._analyze_monthly()
        else:
            return self._analyze_semester()

    def _analyze_weekly(self, week_number: int) -> dict:
        """周报分析（核心功能）"""
        return {
            'overview': self._get_week_overview(week_number),
            'repairs_by_classroom': self._get_repairs_by_classroom(week_number),
            'week_comparison': self._compare_with_last_week(week_number),
            'fault_analysis': self._analyze_faults(week_number),
            'area_analysis': self._analyze_areas(week_number),
            'time_analysis': self._analyze_time(week_number),
            'college_analysis': self._analyze_colleges(week_number),
            'advice': self._generate_advice(week_number),
        }
```

## 附录：分析指标定义

### 核心指标

| 指标 | 计算公式 | 说明 |
|------|----------|------|
| 报修量 | COUNT(records) | 时间范围内的报修总数 |
| 待处理数 | COUNT(status='未处理' OR '处理中') | 当前未完成的报修数 |
| 处理率 | 已处理数 / 总数 × 100% | 报修处理完成比例 |
| 环比增长率 | (本期 - 上期) / 上期 × 100% | 与上一个周期对比 |

### 异常检测规则

| 规则 | 阈值 | 说明 |
|------|------|------|
| 教室反复报修 | >= 5次/学期 | 某教室频繁报修，需检查设备 |
| 故障类型集中度 | > 30% | 某类故障占比过高，需培训 |
| 楼栋故障率异常 | > 40% | 某楼栋故障率过高，需排查 |
| 待处理积压 | > 5件 | 待处理工单过多，需优先处理 |
| 周报修量激增 | > 30% | 较上周增长过多，需关注 |
| 报修高峰 | > 5次 | 某时段报修集中，需增派人手 |
